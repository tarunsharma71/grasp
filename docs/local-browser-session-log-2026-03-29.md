# 本地浏览器操作过程日志（2026-03-29）

## 背景

本次任务目标：

- 在用户当前可见的本地浏览器中打开 `ChatGPT`、`Gemini`、`Grok`、`豆包`
- 向多个页面发送相同问题
- 后续再把统一关键词 `ai agent` 发到四个页面

## 过程里暴露的问题

### 1. 浏览器实例选择越权

- 在用户没有明确确认的情况下，自动从“当前本地浏览器”切到了“可控浏览器 / headless 调试实例”
- 造成“我以为在本地浏览器操作，实际却是在 `127.0.0.1:9222` 的 `HeadlessChrome` 里提问”

影响：

- 违背了用户对操作边界的要求
- 回答中一度把 `headless` 实例误说成“当前本地浏览器”

### 2. 对浏览器上下文的表述不准确

- 把“能看到相同标签页”错误表述成“就是用户当前正在用的那一个浏览器会话”
- 没有先核实 `9222` 对应的是哪个实例、是否 `headless`

影响：

- 误导了用户
- 降低了过程可追溯性

### 3. 默认偏向自动化浏览器路径

- 涉及网页自动操作时，习惯性优先选择更容易验证的自动化路径
- 没有把“由谁决定浏览器类型”放在第一优先级

影响：

- 和用户的使用习惯冲突
- 增加了不必要的切换成本

### 4. 页面状态识别不稳定

- `ChatGPT` 一直卡在 Cloudflare / “请稍候…” 页面
- `Grok`、`Gemini`、`豆包` 在不同阶段的 DOM、输入框、标题都会变化
- 标签页标题在发送后会变化，导致基于固定标题的定位失效

影响：

- 自动化脚本容易误判页面状态
- 已经成功的页面在后续验证时可能被判成“找不到”

### 5. 可见浏览器控制方案切换过晚

- 在确认用户明确要“当前可见本地浏览器”之后，才转向 Windows UI Automation + 前台窗口控制方案
- 前面花了不少时间在 CDP / Playwright / `headless` 路径上

影响：

- 过程绕路
- 用户体验不连续

## 已确认的事实

- `127.0.0.1:9222/json/version` 返回的是 `HeadlessChrome`
- 刚刚前一轮“提问/搜索”确实发生在 `headless` 浏览器里，不是用户当前可见的本地浏览器
- 后一轮 `ai agent` 是通过可见 Chrome 窗口、前台切换标签页、UI Automation / SendKeys 的方式发送的

## 根因总结

1. 缺少“浏览器实例选择必须先确认”的硬规则
2. 缺少“回答前先核实当前控制的是哪一个实例”的检查步骤
3. 现有自动化习惯更偏向 CDP / Playwright，可见浏览器自动化不是默认路径

## 待解决 Plan

### P0：交互规则

- 所有浏览器相关任务先确认实例类型：
  - 当前可见本地浏览器
  - 单独可控浏览器
  - headless 浏览器
- 用户未明确前，不自动切换

### P0：执行前检查

- 真正执行前先记录当前目标实例：
  - 是否可见
  - 是否 headless
  - 连接端口
  - 当前窗口标题
- 回答用户时只能基于已核实的信息描述实例

### P1：可见浏览器操作流程

- 优先准备一条“可见 Chrome 标准流程”：
  - 枚举窗口
  - 枚举标签页
  - 定位输入框
  - 输入文本
  - 发送
  - 轻量验证

### P1：标签页识别增强

- 不只靠固定标题识别页面
- 增加 URL、窗口标题片段、页面可访问名称等多重匹配

### P1：状态提示

- 在每次跨实例前，先明确提示：
  - 当前准备使用哪个浏览器实例
  - 为什么要用它
  - 是否需要用户确认

### P2：失败分类

- 把失败分为：
  - 登录/风控拦截
  - 页面结构变化
  - 浏览器实例选错
  - 权限不足
  - 输入框定位失败

## 这次流程的结论

- 这次最核心的问题不是“网页提问失败”，而是“浏览器实例选择和表述没有先交给用户决定”
- 后续同类任务必须先把实例边界说清楚，再谈自动化执行
